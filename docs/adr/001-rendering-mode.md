# ADR 001 — Rendering mode: `static` with opt-in SSR

## Context

The portal has three route categories:

- **Landing** (`/`) — static marketing content, no per-user data.
- **Search** (`/search`) — static shell; all search interactivity happens in
  Svelte islands on the client. The initial page itself doesn't need per-user
  HTML.
- **Detail** (`/opportunities/[source]/[id]`) — needs to fetch one opportunity
  from the right source API per request. IDs are unbounded (one per grant) and
  data changes, so prerendering every combination is infeasible.

## Decision

Use Astro 5/6's default `output: 'static'` with `export const prerender = false`
on the detail route. Other routes prerender at build time.

## Why not `output: 'server'`?

Would work, but makes the landing page unnecessarily dynamic. `static` with
per-route opt-in SSR is the Astro 5+ replacement for the removed `hybrid` mode
and matches our traffic shape better.

## Consequences

- The landing page is fully cacheable at the CDN edge.
- The detail route runs on Cloudflare Workers via `@astrojs/cloudflare`.
- Deploy target is Cloudflare Workers + static assets (the newer replacement
  for Cloudflare Pages); see `dist/server/wrangler.json` generated by the
  adapter.
